Architecture · Nick's recommendation

How the Career Pivot Agent gets built

One page. No fluff. Read this end-to-end and you should know exactly what we're shipping, in what order, with what stack — and what we're explicitly NOT building in v1.

What Codex would push back on: "You don't need two Claude calls and you don't need to LLM-generate the diagnostic. Curated role-risk data is faster, deterministic, and citeable." See the Compare tab — Codex is right on both counts. The hybrid recommendation incorporates them.

1. What it is (and isn't)

It is

  • A single tool at huva.io/pivot that takes a job role + CV and returns a personalized AI-pivot plan.
  • A self-contained Next.js app, deployed on Vercel, routed under huva.io via rewrite.
  • Bilingual (EN + AM) from day one. Same code, two language packs.
  • Anchored in real Anthropic Economic Index research — every diagnostic number cites a source.

It is not (in v1)

  • Not a multi-tenant SaaS. No accounts, no logins, no payments.
  • Not integrated into the existing huva.io codebase. Its own repo, mounted via routing.
  • No save/share URLs in v1 — output lives in the browser tab.
  • No CV upload reading on our backend — PDF goes straight to Claude (native support).

Architecture · Codex's recommendation

Build the simplest version that preserves the wow

One page. One AI call. No database. No accounts. No saved results. Strip everything we don't need to keep ~85–90% of the user-perceived value.

Nick's reviewer's note: Codex is right on the two big calls — local deterministic diagnostic and one Claude call. I disagree on dropping abuse prevention; for a public tool that costs money per use, that's an open door. The Compare tab has the resolved version that takes Codex's simplifications and keeps the one safety net.

1. Goal

Build a simple, high-value AI career pivot tool for HuVa that shows users how exposed their current role is to AI, then gives them a personalized 30-day pivot plan based on their CV.

2. Simplified architecture

huva.io/pivot
  ↓
User enters role
  ↓
Instant local diagnostic     ← no API call · curated data · cited
  ↓
User pastes CV or uploads PDF
  ↓
One Claude API call          ← only call in the entire flow
  ↓
Structured pivot plan renders in browser
  ↓
User downloads PDF / clicks HuVa CTA

The diagnostic is a static lookup. The personalized plan is the only LLM-generated piece. That's it.

3. What we keep

  • Strong hook: "Will Claude take your job?"
  • Bilingual EN/HY experience
  • Role-based AI exposure diagnostic
  • CV-based personalization
  • 5 ranked pivot paths
  • 30-day practical plan
  • LinkedIn positioning
  • 3 portfolio projects
  • Identity card
  • Browser PDF export
  • One strong HuVa Academy CTA

4. What we cut from v1

  • No login
  • No payments
  • No dashboard
  • No saved result pages
  • No Supabase
  • No personalized share links
  • No second diagnostic API call
  • No complex backend PDF generation

5. Why this is better

Keeps ~85–90% of the user-perceived value while removing most of the technical complexity. Result:

  • Faster to build
  • Cheaper to run
  • Easier to QA in Armenian and English
  • Less fragile
  • Easier to launch as a campaign tool
  • Still powerful enough to convert users into HuVa Academy leads

6. Recommended v1 scope

Standalone Next.js app at /pivot with:

  • Local role-risk data for common roles
  • One /api/plan endpoint using Claude
  • Zod validation for structured output
  • EN/HY copy files
  • HuVa visual branding
  • Print CSS for "Download PDF"
  • Simple analytics: viewed diagnostic, submitted CV, generated result, clicked CTA

7. Ani review needed

  • Armenian wording of the core promise
  • Whether the AI-risk framing feels motivating, not scary
  • Whether the HuVa CTA feels natural after the result
  • Whether the 30-day plan feels believable and useful
  • Whether the identity card is something people would actually share

Recommendation

Build this optimized v1 first, launch quickly, then add saved/shareable results only if people actually use it.

2. The hybrid · what we should actually build

Recommended v1 (hybrid)

Take Codex's simplifications. Keep Nick's one safety net. Result is sharper than either alone.

  • Local deterministic diagnostic from curated role-risk data (Codex)
  • One Claude call total for the personalized plan (Codex)
  • Zod validation on the Claude response (Codex)
  • Cloudflare Turnstile on the generate endpoint (Nick — non-negotiable)
  • Streaming via SSE for the plan generation UX (Nick — keep if time allows; otherwise just await)
  • Bilingual EN + AM via next-intl, locale-prefixed routes (both)
  • Print CSS for PDF, no server-side renderer (both)
  • Vercel + huva.io rewrite, no Supabase in v1 (both)

Build budget: 5–7 hours focused. Target launch in 2–3 days.

Run cost: ~$30 / 1k generations (single Sonnet call, ~3k in / ~3k out).

3. The role-risk data file (the new piece)

This is the one new artifact the hybrid needs. A static JSON / TS file with curated diagnostics for ~25 common roles. Build this once, it powers every diagnostic instantly without any LLM cost.

// lib/role-risk.ts
export const ROLE_RISK = {
  "marketing-manager": {
    exposure: 47,
    aug: 38, auto: 47, safe: 15,
    blurb: { en: "...", hy: "..." },
    tasks: [
      { tag: "auto", label: { en: "...", hy: "..." }, text: { en: "...", hy: "..." } },
      // ...
    ],
    cite: "Anthropic Economic Index, Feb 2026"
  },
  "software-engineer": { ... },
  "customer-service-rep": { ... },
  // ~25 roles total for v1 launch
}

Fallback for unknown roles: a generic "we don't have specific data for this role yet — here's the general augment-vs-automate framing" diagnostic that still feels personal.
